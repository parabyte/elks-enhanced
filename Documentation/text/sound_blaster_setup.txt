ELKS Sound Blaster and MAD16 audio setup
========================================

Sound Blaster compatible audio
------------------------------

Enable the ISA DMA audio driver with CONFIG_CHAR_DEV_DSP.  The driver exposes
/dev/dsp and /dev/audio as OSS-style character devices using major 5, minor 0.
It supports 8-bit ISA DMA playback on DMA channel 1 or 3.  Mono unsigned
8-bit PCM is the default format; CONFIG_SB_STEREO adds SB Pro style stereo
playback support when the hardware supports it.

The compile-time defaults are:

  CONFIG_SB_PORT    Sound Blaster base I/O port, normally 0x220
  CONFIG_SB_IRQ     Sound Blaster IRQ, commonly 5 or 7 on XT-class systems
  CONFIG_SB_DMA     8-bit DMA channel, 1 or 3
  CONFIG_SB_BOUNCE  DMA bounce buffer size

/bootopts can override the compiled port, IRQ, and DMA without rebuilding:

  sb=0x220,5,1

The port may also be written as decimal:

  sb=544,5,1

To build one kernel that can also run on machines without a sound card, leave
CONFIG_CHAR_DEV_DSP enabled and disable probing at boot:

  sb=off

sb=no and sb=0 are accepted aliases for sb=off.

OPTi 82C929 / MAD16 Pro cards
-----------------------------

OPTi 82C929 based MAD16 Pro cards can be used by forcing the card into Sound
Blaster compatible mode, then using the normal ELKS Sound Blaster driver.
The ELKS driver does not use the Windows Sound System codec path.

The kernel leaves OPTi/MAD16 registers untouched by default so the same SB
driver can safely boot on real Sound Blaster cards.  To enable the in-kernel
MAD16 bring-up, keep the normal SB route in sb= and add mad16=on:

  sb=0x220,5,1
  mad16=on

mad16=on programs the card from the active SB route.  If no sb= line is present
it uses CONFIG_SB_PORT, CONFIG_SB_IRQ, and CONFIG_SB_DMA.  The built-in profile
matches the XT-safe mad16cfg defaults: WSS, CD-ROM, IDE, and MPU decode are
disabled, SB Pro compatible mode is selected, and only the Sound Blaster path is
left for /dev/dsp.

If a board needs the MAD16 route forced explicitly, use either:

  mad16=5,1
  mad16=0x220,5,1

The first form forces MAD16 IRQ/DMA and follows the active sb= port.  The second
form forces MAD16 SB port, IRQ, and DMA.  Valid routes are SB port 0x220 or
0x240, IRQ 5 or 7, and DMA 1 or 3.  Keep forced mad16= routing matched to sb=,
for example sb=0x220,5,1 with mad16=0x220,5,1.

Use mad16=off, or omit mad16= entirely, for real Sound Blaster cards and other
SB-compatible hardware that must not receive OPTi control-window writes.

Userland setup with mad16cfg
----------------------------

Build CONFIG_APP_MAD16CFG and enable /dev/port read and write support with:

  CONFIG_CHAR_DEV_MEM_PORT_READ
  CONFIG_CHAR_DEV_MEM_PORT_WRITE

Then configure the card from ELKS:

  mad16cfg defaults

or explicitly:

  mad16cfg set mode sb sb_port 0x220 sb_irq 5 sb_dma 1

mad16cfg accepts only XT-safe routes: SB IRQ 5 or 7 and DMA 1 or 3.  It forces
WSS and CD-ROM/IDE decode off.  Keep /bootopts sb= matched to the card setting,
for example sb=0x220,5,1.

Basic use
---------

The driver accepts simple OSS-style writes to /dev/dsp or /dev/audio.  The
most portable format is unsigned 8-bit mono PCM at a modest sample rate.
Applications that assume 16-bit samples, DMA channels above 3, or PCI/AC97
style audio will need conversion before playback.

For a diagnostic boot, start with the most common XT-friendly settings:

  sb=0x220,5,1

If the machine has another ISA card using IRQ 5, try IRQ 7:

  sb=0x220,7,1

Troubleshooting
---------------

If the machine boots but audio is silent, first confirm that sb= in /bootopts
matches the card jumpers or mad16cfg settings.  A wrong IRQ can make playback
appear to hang until the DMA buffer times out.

If a MAD16 card conflicts with an IDE or CD-ROM interface, use mad16=on or
mad16cfg.  Both paths intentionally disable the MAD16 WSS, IDE, CD-ROM, and MPU
decode paths so the card behaves like a simple Sound Blaster compatible device.

If the same kernel image is shared between machines, use sb=off on systems
without an ISA sound card so the driver does not probe missing hardware.
